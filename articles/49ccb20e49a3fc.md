## 組版システム

組版システムには [Typst] を利用しました。

Typstは、一言で言うと、爆速でコンパイル可能なMarkdown風の文法を持つLaTeXです。

<!-- textlint-disable -->
文法がLaTeXとかなり違うため、最初は戸惑うかもしれませんが、このリアルタイムコンパイルは一度使うと戻れなくなる中毒性があります。タイプした瞬間にプレビューが更新されるのは非常に快適です。コンパイルに10秒以上待たされるLaTeXにはもう戻れません。
<!-- textlint-enable -->

https://typst.app/

有志による日本語訳：

https://typst-jp.github.io/docs/

### コードブロック

コードブロックの装飾にはCodlyを利用しました。

Codlyは、Typst用のコードブロック装飾パッケージで、行番号の表示や、いい感じにインデントする機能などを提供してくれます。

Typstのデフォルトのコードブロックはかなり質素な見た目なので、こういった装飾パッケージを併用するのがおすすめです。

https://typst.app/universe/package/codly/

### 疑似コード

疑似コードの記述にはalgorithmicを利用しました。

algorithmicは、Typst用の疑似コード記述パッケージで、Typst内で疑似コードが書けるようになります。LaTeXの [algorithmicx](https://ctan.org/pkg/algorithmicx) のTypst版みたいなイメージです。

特にこだわりはないですが、一番メンテナンスされてそうだったので選びました。

https://typst.app/universe/package/algorithmic/

## CI/CD

CIはGitHub Actionsを利用しました。

実行環境は、[devcontainers/ci](https://github.com/devcontainers/ci) を使いDevcontainerをそのまま利用しました。
こういった点でも、Devcontainerは作っておくと便利です。

https://github.com/devcontainers/ci

次に、実際に利用したワークフローファイルを参考までに示します：

```yaml
# .github/workflows/ci.yml

name: CI

on:
  push:
    branches:
      - main
  pull_request:

permissions:
  contents: read
  packages: write

jobs:
  ci:
    runs-on: ubuntu-24.04
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Free disk space
        run: |
          df -h
          sudo rm -rf /usr/share/dotnet /usr/local/lib/android /opt/ghc /opt/hostedtoolcache /usr/local/share/powershell
          sudo apt-get clean
          docker system prune -af
          df -h

      - name: Cache compiler build outputs
        uses: actions/cache@v4
        with:
          path: |
            compiler/.build.dir
            compiler/install
          key: ${{ runner.os }}-compiler-build-${{ hashFiles('compiler/CMakeLists.txt', 'compiler/**/CMakeLists.txt', 'compiler/**/*.cmake', 'compiler/Makefile') }}-v1
          restore-keys: |
            ${{ runner.os }}-compiler-build-

      - name: Create necessary dirs
        run: |
          mkdir -p .devcontainer/.codex
          mkdir -p .devcontainer/.claude

      - name: Run in devcontainer
        uses: devcontainers/ci@v0.3
        with:
          configFile: .devcontainer/devcontainer.json
          imageName: ghcr.io/${{ github.repository }}-devcontainer
          cacheFrom: ghcr.io/${{ github.repository }}-devcontainer
          push: always
          runCmd: |
            npm ci

            # Build
            just build-compiler || { echo "Build failed"; exit 1; }

            # Unit Tests
            just test-compiler || { echo "Unit Tests failed"; exit 1; }

            # E2E Tests
            SKIP_BUILD=1 just test-e2e || { echo "E2E Tests failed"; exit 1; }
```

なお、普通に実行した際にディスク容量不足によるエラーが発生したため、不要なファイルを削除するステップを追加しています。

````md
## 全体的な注意事項

- **口語体** で記述すること
- 初出単語は太字、かっこで説明をつけること

  ```typ
  *制御フローグラフ*（Control Flow Graph、以下CFGと呼ぶ）
  ```

- self-contained にすること

## 文章の書き方（よくある修正）

### 段落内の構造

- 尾括法 → 頭括法 ＝ 先頭にトピックセンテンスを書く。

### 文章

- 非常に長い文 → 主題ごとの短い文に分ける。ある程度の長さは許容するが、極めて長い文は分割する。
- 羅列している長い文 → 箇条書きにする 。
- 箇条書きの上下に分かれた文 → 箇条書きの上にまとめる。
- 指示代名詞（これ、その、… など） → 必要ない場合はトル ／ 指示先の名詞に置き換える（本当に必要なときだけ指示代名詞を使う）。
- 読点（、） → 主題（主語）の直後には、点（、）を入れる。

### 用語

下記は、『公用文作成の要領』(PDF) 及び 日本工業規格 JIS Z 8301（規格票の様式及び作成方法） に準じている。

目的：誤読を避ける。
方針：一つの用語は、一つの意味だけに用いる。

- 目的：ため → ため
- 理由：ため → ので ｛から｝
- 理由：そのため、 → したがって、
- 比較：より → より
- 原因：より → よって
- 起点：より → から
  （“より”が比較か原因かが判読できない例： “○○ 処理により効果がある” → 比較として解釈する。）
- 限定：文語的 のみ → だけ
- 順接：が、 → 。
- 逆接：が、 → 。しかし、
- 道具：文語 にて → で
- 原因：で → によって
- 並立：で → であって
- 可能：を……られる → が……られる
- 提示：通り → とおり（助数詞の場合は漢字： “3 通り”）
- 提示：ように → とおりに（厳密な場合）
- いい → よい ｛良い｝

### 用字

- 事 → こと
- 物 → もの
- 等 → など（法律以外の場合）
- 持つ → もつ（手で“持つ”以外の場合）
- と言う → という（口で“言う”以外の場合）
- 出来る → できる
- 無い → ない
- 関わらず → かかわらず（× 拘らず）

### 送り仮名

- 複合動詞：組合せる → 組み合わせる etc.
- その名詞：組み合わせ → 組合せ etc.
````
